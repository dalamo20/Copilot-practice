# Rules: Governance Artifacts

These rules apply whenever you are reading or editing files under `governance/`.

## Change process (mandatory)

Before modifying any file in `governance/`:

1. Run `/self-audit` to record a baseline score.
2. Make the change in a **branch**, not on main.
3. Run `/self-audit` again after the change.
4. Compare the two self-audit reports. Explain any score changes in the commit message.

## governance/scoring-standard.md

- This file is the **single source of truth** for all scoring weights and rubrics.
- Do not add scoring logic or weight declarations anywhere else in the project.
- Changes to rubric criteria (0–5 levels) require re-running golden calibration tests.
- Never remove a rubric level — only refine the criteria text.
- Any edit to this file is blocked by the pre-tool-use hook. Follow the change process above.

## governance/references/anti-patterns.md

- Each anti-pattern requires: a unique ID (AP-N), a pattern example, a risk explanation,
  and a fix.
- Anti-pattern additions are **additive only**. Do not renumber existing IDs — they are
  cross-referenced in audit reports by ID.
- Any edit to this file is blocked by the pre-tool-use hook. Follow the change process above.

## governance/output/

- Save reports using the naming convention in `governance/output/README.md`.
- Never delete self-audit reports. They are the framework's health record.
- Keep at least the 3 most recent reports per asset type.
- `.change-log.txt` is auto-generated by the PostToolUse hook — do not edit manually.

## governance/VERSION

- Bump with every PR that changes framework behavior.
- Semantic versioning: MAJOR for breaking changes to scoring or verdicts, MINOR for new
  features, PATCH for fixes and clarifications.
